Git 版控 git format-patch 和 git bundle 的差異

發表於 2021-05-29 分類於 工作雜記

情境說明

舉例情境，公司有 Sam 和 Jack 兩位開發者，用公司內部 Git 版控儲存庫進行開發。

簡述版控流程，開發者先在自己的分支開發，再將結果合併回 Develop 分支驗測功能，驗證無誤再透過 PR 程序合併至 master 分支上線。

公司因為安全性議題，內部版控儲存庫不提供任何管道給外部網路連線使用

如下圖，假設公司在 Commit ID e3251c8 時開始實施 Work From Home，從這裡開始就無法再連線至公司內的遠端儲存庫。

Work From Home 後，Sam 修改本地 Develop_Sam 分支的 README.md 檔案內容，且合併回本地的 Develop 分支

同時，Jack 也在修改本地 Develop_Jack 分支的 README.md 檔案內容，且合併回本地的 Develop 分支

前兩張圖可看到 Sam 與 Jack 在本地 Develop 分支的版控記錄已經不一致了，雙方的 README.md 檔案內容如下

接下來 Sam 和 Jack 要同步彼此的開發進度，但是連不到公司的版控儲存庫了，此時有以下幾種選擇

可用的做法

  1. 用私有 Git 遠端儲存庫同步彼此的版控記錄 (適合不守規定的人偷偷用)

    • 此做法只能解決開發者間的版控問題，但仍然無法解決公司內部儲存庫的上版，而且可能違反規定

  2. 將 Source Code (包含 .git) 目錄通通打包傳給其他開發者或公司 (你瘋了嗎？ 😆)

  3. 用 git bundle 產生版控記錄檔 (★ 推薦)

    • 這是一個 binary 的檔案，裡面除了有檔案的修改內容，也包含 commit 版控記錄

    • 本地端可透過 git pull 將記錄檔的版控內容 pull 至本地分支，並且保持原有的開發版控記錄，當然也可以送入公司同步版控記錄

  4. 用 git format-patch 產生內容修補檔 (只適合交付修補程式用，不適合開發用)

    • 開發者可套用修補檔以更新彼此修改過的檔案內容，也可送入公司上版，但是會汙染版控記錄，最終大家的版控記錄都會不一致

    • 修補檔記錄的是 git diff 的差異內容，不會保留版控記錄，套用修補檔時，在背後是使用 git commit 方式處理，會產生新的版控記錄 (commit hash)

    • 因為版控記錄脫鉤了，不僅造成溝通上的困難，將來解封回公司連版控儲存庫時，也會再痛一次要解決大量合併衝突的問題

    • 因此 git format-patch 只適合交付修補程式用，不適合開發用

為了節省大家寶貴的時間，先說結論

  • 如果你是守規定的開發人員

    • 用 git bundle 產生 Develop 分支的版控記錄檔，並交付給其他開發者同步 Develop 分支的版控記錄
    • 如果自己有修改過檔案，在 git pull 合併完後要再做一份 git bundle 給其他開發人員同步

  • 如果你是無視規定的開發人員

    • 平時用私有 Git 儲存庫進行開發

    (安全性要做好，否則 Source Code 外流就……會變成 Open Source 😆 )

    • 只有最後要進公司上版時，才用 git bundle 產生 Develop 分支的版控記錄檔丟到公司內做 git pull 即可

接下來說實作，有興趣的人請繼續看下去

實作

私有 Git 遠端儲存庫

這個做法就不說明了，有免費的、付費的、私有自架的…路很多條，任君挑選。

一樣為了節省大家寶貴的時間，先說個人推薦的 git bundle 實作方式

用 git bundle 產生版控記錄檔 (★ 推薦)

先把時光回朔到公司開始實施 Work From Home，並且 Sam 和 Jack 都在自己的本地分支修改檔案的狀態

Sam 用 git bundle 產生從 e3251c8 到 Develop 分支的版控記錄檔 Develop_FromSam，並交付給 Jack

會選擇 e3251c8 是因為 Jack 也有這個 Commit，可做為雙方 Develop 分支共同的基準點

Jack 收到檔案後，先用 git bundle verify 檢查檔案內容與裡面的分支

可看到此檔案內有 Develop 分支的版控記錄，並且本地端必須要有 e3251c8 這個 Commit 才可 git pull

把 Develop_FromSam 想像成是一個遠端儲存庫，裡面有一個 Develop 遠端分支
Jack 現在想把遠端 Develop 分支 Pull 到本地 Develop 分支
流程和平常 Pull 一樣先 checkout 到本地 Develop 分支，然後 git pull 遠端儲存庫的 Develop 分支

此時發生衝突進入 MERGING 狀態了，因為遠端 Develop 分支有 Sam 的修改

解決衝突，圖內 Develop_FromSam 檔案可無視，等 git pull 完就可以刪掉了

因為 Jack 有處理過衝突，所以合併好的本地 Develop 也輸出版控記錄給 Sam
要用 Sam 也有的 Commit ID e3251c8 做為基準點，並將產生的 Develop_FromJack 檔案交付給 Sam

Sam 收到後一樣先確認一下版控記錄檔內是否有 Develop 遠端分支和必要的 Commit ID e3251c8
沒問題就直接 git pull 到本地 Develop 分支，因為 Jack 已經解完衝突了，所以本地 Develop 分支直接 Pull 就完成了
記得將 Develop_FromJack 檔案刪除

現在 Sam 和 Jack 的版控狀態一模一樣，如果要送進公司上版，也可以直接把 Develop_FromJack 檔案傳進公司

再遠端遙控公司的電腦透過 git pull 把 Develop 併進去上版

因為大家的版控記錄都是維持同步的狀態，將來解封連回公司版控儲存庫，也是無縫銜接。

用 git format-patch 產生內容修補檔 (不建議開發用)

先把時光回朔到公司開始實施 Work From Home，並且 Sam 和 Jack 都在自己的本地分支修改檔案的狀態

Sam 使用 git format-patch 產生 e3251c8 到 4df1ae8 之間的內容修補檔

產生了兩個修補檔，並交付給 Jack

Jack 收到 Sam 交付的內容修補檔，使用 git am 將修補內容 commit 至本地的 Develop 分支

因為雙方 Develop 分支的 README.md 都有修改，發生衝突了

聰明的 Sam 早早寫完先交付就沒事了，寫得慢的 Jack 就要解衝突，陷入被越拖越慢的惡性循環 😆

見圖右上方，目前狀態進入 Develop|AM 解衝突模式了
倒楣的 Jack 先看看是什麼造成衝突

把造成衝突，缺少的內容補進 README.md，並存檔

把修改的結果加入 git 索引

問題排除，繼續執行 AM

第二個修補檔也發生衝突了，Jack 在心裡咒罵 Sam

再來看這次的衝突原因

再次把造成衝突，缺少的內容補進 README.md，並存檔

把修改的結果加入 git 索引

問題排除，繼續執行 AM

見圖右方 Develop|AM 狀態結束，總算合併完成了

現在 Jack 本地 Develop 分支多了兩個 commit 將 Sam 修改的內容併進來了

等等，做到這邊，有人注意到 Sam 還沒拿到 Jack 合併後的結果嗎？

所以 Jack 要再 git format-patch 產生 Develop 分支的修補檔回去給 Sam？

Sam 將修補檔合併到自己本地的 Develop 分支又產生新的 commit，版控記錄越來越不一致…

在可以連到遠端儲存庫之前，大家的 Commit ID 都對不起來，有時會造成溝通上的困難，即使連到遠端儲存庫了，也免不了要再解合併衝突

由此看來 git format-patch 並不適合開發過程的版控交換用途，他的重點是檔案的修補，做為交付修補程式的一種方式，而非版控記錄

以上是小弟個人的理解和實務操作經驗分享，如果有錯誤的地方還請不吝指點，謝謝 😊
