Git 離線同步版控記錄 git format-patch 和 git bundle 的差異

發表於 2021-05-29 分類於 工作雜記

模擬情境

公司有 Sam 和 Jack 兩位開發者，用公司內部 Git 版控儲存庫進行開發。

簡述版控流程，開發者先在自己的分支開發，再將結果合併回 Develop 分支驗測功能，驗證無誤再透過 PR 程序合併至 master 分支上線。

公司內部版控儲存庫不提供任何管道給外部網路連線使用，但有開放 VPN 可遠端連線進去控制公司內部的電腦。

如下圖，假設公司在 Commit ID e3251c8 時開始實施 Work From Home，從這裡開始就無法再連線至公司內的遠端儲存庫。

Work From Home 後，Sam 修改本地 Develop_Sam 分支的 README.md 檔案內容，且合併回本地的 Develop 分支。

同時，Jack 也在修改本地 Develop_Jack 分支的 README.md 檔案內容，且合併回本地的 Develop 分支。

前兩張圖可看到 Sam 與 Jack 在本地 Develop 分支的版控記錄已經不一致了，雙方的 README.md 檔案內容如下

接下來 Sam 和 Jack 要同步彼此的開發進度，但是連不到公司的版控儲存庫了，此時有以下幾種選擇

可用的方法

  1. 在私有 Git 遠端儲存庫同步彼此的版控記錄 (公司沒有禁止的情況適用)

    • 此做法無法解決上版至公司內部儲存庫的問題，只能解決開發者間的版控同步問題

    • 公司不一定允許 Source Code 存放在外部的私有遠端儲存庫

  2. 將 Source Code (包含 .git 目錄) 通通打包傳給其他開發者或公司 (你瘋了嗎？ 😆)

  3. 用 git bundle 產生版控記錄檔 (★ 推薦)

    • 會產生一個 binary 檔案，裡面有檔案的修改內容，也包含 commit 版控記錄

    • 本地端可透過 git pull 將記錄檔的版控內容 pull 至本地分支，並且保持開發的版控記錄，也可以送入公司同步版控記錄

  4. 用 git format-patch 產生內容修補檔 (只適合交付修補程式用，不適合開發用)

    • 開發者可套用修補檔以更新彼此修改過的檔案內容，也可送入公司上版，但是在開發過程中，每個人的版控記錄會不一致

    • 修補檔記錄的是 git diff 的差異內容，不會保留版控記錄，套用修補檔時，在背後是使用 git commit 方式處理，會產生新的版控記錄 (commit hash)

    • 版控記錄脫鉤了，不僅造成溝通上的困難，將來解封回公司連版控儲存庫時，很可能要再面臨一次大量衝突合併的問題

    • 因此 git format-patch 只適合交付修補程式用，不適合開發同步版控用

為了節省大家寶貴的時間，先說結論

  • 如果公司禁止使用私有儲存庫

    • 用 git bundle 產生 Develop 分支的版控記錄檔，交付給其他開發者同步 Develop 分支的版控記錄

    • 如果開發者的修改已告一段落，在 git pull 合併完別人的版控記錄後，就再做一份 git bundle 給其他開發人員同步，可把這步驟當成原本要 git push 的替代操作

  • 如果公司同意使用私有儲存庫 (★ 個人認為這是最佳狀況)

    • 平時用私有 Git 儲存庫進行開發

    (安全性要做好，否則 Source Code 外流就……會變成 Open Source 😆 )

    • 在開發完成要上版到公司時，先用 git bundle 產生 Develop 分支的版控記錄檔，再丟到公司內做 git pull 即可 (前提是公司有開放 VPN 可遠端連線內部電腦作業的情況)
